# 2026-04-28 세션 로그

## 요약

API 호출 없이 UX 흐름(타이핑 → 단어 → 확정 → mood)을 빠르게 테스트하기 위한 `MOCK_MODE` 플래그를 추가했다. 로컬에서만 켜고 배포 시 Actions가 자동으로 `false`로 고정 주입한다.

## 주요 변경

- `config.js` (로컬): `MOCK_MODE = true` 플래그 추가. `.gitignore`되어 있어 커밋되지 않음.
- `config.example.js`: `MOCK_MODE = false` 라인 추가, 주석으로 토글 안내.
- `index.html`:
  - `getMockResponse(layer, confirm)` 함수 추가 (callGemini 직전).
    - 레이어별(`장소`/`사람`/`사건`/`유물`) 파트너 단어 풀.
    - `confirm=true`면 80% 확률 agree, 20% disagree.
    - 일반 턴은 `turnCount >= 2`에서 20% 확률 `propose_end`.
    - mood 4종(`primary_color`/`secondary_color`/`font_tone`/`keywords`) 랜덤 선택.
  - `callGemini` 진입부에 mock 분기: `MOCK_MODE` 켜져 있으면 800–1400ms 딜레이 후 `getMockResponse` 즉시 반환.
- `.github/workflows/deploy.yml`: config.js 주입 단계에서 `MOCK_MODE = false`도 함께 기록되도록 변경. 라이브 환경은 mock으로 빠질 위험 없음.
- `CLAUDE.md`: API 키 관리 규칙 섹션에 mock 모드 한 줄 추가.

## 결정 사항

- **mock 분기는 `callGemini` 함수 안쪽**에 두었다. 호출부(`submitWords`/`handleConfirm`) 두 곳을 건드리지 않아도 되고, 응답 데이터 형식도 한 곳에서만 보장하면 되어 유지가 단순.
- **실제 키와 mock 플래그는 같은 `config.js`에 둔다.** 둘 다 로컬 전용이고 같은 시점에 토글되므로 별도 파일로 분리할 동기가 없다.
- **deploy.yml에서 `MOCK_MODE=false`를 강제 주입**하는 방식을 택했다 (사용자 지시). 실수로 mock=true 상태로 라이브에 흘러들어갈 통로 자체를 없애는 안전판.
- **`partner_words`는 confirm 케이스에서도 채워서 반환.** 실제 게임 로직은 `CONFIRM_PARTNER_AGREE/DISAGREE` 풀에서 따로 뽑기 때문에 mock의 `partner_words`는 사용되지 않지만, 응답 스키마 일관성 차원에서 유지.
