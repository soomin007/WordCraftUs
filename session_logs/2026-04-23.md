## 요약

플레이테스트 피드백을 반영해 게임 언어를 "단어 전용"으로 재정립. 규칙 안내 화면, 단어 기반 파트너 응답, 세션마다 쌓이는 ASCII 아트 배경을 도입. 기본 모델을 flash-lite로 전환하고, 재시작 시 화면 잔여물이 남던 버그도 함께 수정.

## 주요 변경 (`index.html`)

### 1) 재시작 버그 수정 (`new-world-btn`)
- `chatPanel`/`worldLayersEl`/`session-end-choice`/`ascii-bg-container` 전부 초기화
- 세션 종료 후 숨겨뒀던 `input-area` / `input-hint` 복원, `wordInput.disabled = false`
- 이전엔 state만 리셋해서 이전 세계의 말풍선·WORLD 레이어·세션 종료 버튼이 새 플레이에 끌려 들어감

### 2) 규칙 안내 화면 (`rules-screen`)
- 흐름: `intro-screen` → `rules-screen` → `matching-screen`
- 공지 톤 박스(`NOTICE` 라벨 + 본문 + "준비되셨나요?" 질문 + `준비됐어요` 버튼)
- 본문은 `white-space: pre-wrap`으로 들여쓰기/줄바꿈 유지. 내용은 "단어로만 소통", "장소→사람→사건→유물", 각 단계 동작 설명
- `new-world-btn`에서는 규칙을 이미 봤으므로 `rules-screen`을 건너뛰고 `startMatching()`으로 직행 (의도적 선택)

### 3) 기본 모델 교체 (`MODELS`)
- `['gemini-2.5-flash-lite', 'gemini-2.5-flash']` — lite가 기본, flash가 폴백
- 이유: 사용자 요청. lite가 무료 티어에서 훨씬 덜 붐비고 세계 생성 수준에 충분. 품질 저하가 체감되면 flash가 받아줌

### 4) 파트너 응답을 단어로 (`partner_words` / `greeting_words`)
- **Gemini JSON 스키마 전면 변경**:
  - `partner_message` 삭제 → `partner_words: [3~5단어]`
  - `greeting` 삭제 → `greeting_words: [2~4단어]`
  - `ascii_art: "30자×15행 문자열"` 추가
  - `narrative` / `mood` 유지
- **프롬프트 전체 한국어화** (`buildPrompt`, `buildGreetingPrompt`)
  - "단어로만 소통" 규칙 명시, 좋은 예/나쁜 예 각각 두 세트
  - `priorAscii` 블록 주입: 이전 ASCII를 프롬프트에 넣어 "분위기를 이어서 발전시키라"고 지시
  - `PERSONAS[name].tone`과 `playerProfile`은 그대로 주입 유지
- **렌더링 변경**:
  - `addPartnerMsg`/`addMyMsg` → `addPartnerWords`/`addMyWords`. 둘 다 `formatWordList(words)` = `["w1", "w2"]` 형태로 출력
  - 파트너 말풍선도 플레이어 말풍선과 동일 스타일(테두리·JetBrains Mono)로 통일, 정렬만 좌/우로 대칭
  - narrative는 채팅 패널에서 제거 → WORLD 패널(`renderWorld`)에만 노출
  - 세션 종료 말풍선도 단어 배열(`['계속', '할까요']`, `['다', '만들었네요']`)
- **`NEXT_PROMPTS` 상수 삭제**: 2~4레이어 진입 시 파트너가 먼저 문장으로 유도하던 라인 제거. 다음 레이어 진입 후 플레이어 입력이 세션 시작점
- `tryJsonOnce` 검증 키도 `partner_words`로 교체

### 5) ASCII 아트 배경 누적 (`ascii-bg-container`, `addAsciiLayer`)
- `<div id="ascii-bg-container">`를 `<body>` 첫 자식, `position: fixed; inset: 0; z-index: 0`
- `.screen`에 `z-index: 1` 부여, `.chat-panel { background: rgba(13,13,13,0.85) }`, `.world-panel { background: rgba(13,13,13,0.60) }`, 두 패널 `position: relative; z-index: 1`
- 세션마다 새 `.ascii-layer` div를 누적. `color = mood.primary_color` 고정(레이어 생성 시 mood), `font-size`는 `asciiFontSizePx()`로 화면 꽉 차게 자동 계산 (30열 × 0.6 + 15행 비교)
- opacity 규칙: 최신 1.0, 바로 이전 0.35, 그 이전 0.2, 더 이전들 0.1. 새 레이어 추가될 때마다 `updateAsciiOpacities()`가 전체 재계산
- `transition: opacity 2s ease, color 2s ease`로 페이드인·색전환 부드럽게
- `window.resize`에서 `updateAsciiLayoutAll()`로 폰트 크기 재계산
- `new-world-btn`이 컨테이너를 비워 잔여 레이어 제거

## 결정 사항

- **단어로만 소통을 게임 언어로 격상**: 이전 버전은 파트너가 짧은 문장("오, 좋은데요?")을 뱉어 플레이어의 단어 입력과 비대칭했음. 이제 양측 모두 단어 배열로 말풍선 형태가 같아 시각적으로 '같은 규칙을 따르는 두 사람'처럼 보임
- **narrative는 채팅에서 제거, WORLD 패널 전용**: 채팅에 긴 묘사가 섞이면 단어 리듬이 깨짐. 묘사는 세계의 '결과물'로서 WORLD에만. 채팅은 인터랙션, WORLD는 세계. 역할 분리가 명확해짐
- **기본을 lite로 바꾸면서 프롬프트를 한국어로 재작성**: flash보다 instruction following이 약한 lite에 긴 영어 프롬프트를 주면 지시를 흘림. 한국어로 짧고 명시적으로 다시 쓰고, good/bad 예시를 모든 제약에 붙여 few-shot 유도
- **ASCII 아트를 "누적" 방식으로**: 교체가 아니라 겹치기. 초반엔 드문드문, 4세션 후에는 네 층이 겹쳐 빽빽해지는 시각적 성장 = 세계가 레이어로 쌓인다는 개념의 직관적 표현. 최신이 가장 진하고 이전 것이 그림자로 남는 opacity 차등은 플레이어가 '어디까지 만들었는지'를 스스로 느끼게 함
- **ASCII 레이어 color는 생성 당시 mood 고정**: 매번 최신 mood로 덮으면 네 층이 다 같은 색이 되어 누적 효과가 안 보임. 각 레이어가 그 시점 분위기를 간직하고, 전체는 겹쳐진 색 팔레트가 됨
- **패널 반투명: 채팅 0.85, WORLD 0.60**: 채팅은 글자 가독성이 최우선이라 더 불투명. WORLD는 세계 묘사라 배경이 더 비쳐 세계와 묘사가 한 화면에서 녹아 보이도록. 스펙 그대로
- **`new-world-btn`에서 `rules-screen` 스킵**: 규칙은 처음 오는 사람을 위한 onboarding. 새 세계 만들기를 누를 때는 이미 규칙을 아는 사용자. 재노출은 귀찮음만 늘림
- **폰트 크기 자동 계산에서 width/height 중 작은 값 채택**: 가로 기준 계산값이 세로 넘치면 잘리므로 안전한 쪽을 택함. 모바일 세로 화면에서도 30×15가 화면을 벗어나지 않음

## 미해결 / 다음 할 일

- **ASCII 30×15 스키마를 lite가 얼마나 지킬지 미검증**: 프롬프트에 "정확히 30자 × 15행"을 명시하고 이전 ASCII를 예시로 제공했지만, 행 길이가 들쭉날쭉할 가능성. 실플레이에서 깨지면 클라이언트 측 normalize(패딩/크롭) 추가 필요
- **ASCII에 줄바꿈 특수문자가 정확히 `\n`으로 올지**: JSON string에서 `\n`은 OK지만 모델이 `\\n`(백슬래시+n)로 쓸 수도 있음. 실측 후 재파싱 로직 고려
- **단어 기반 대화가 "대화처럼 느껴지는지" 실플레이 감각 체크**: 단어 배열만 오고 가면 오히려 기계적으로 읽힐 위험. 말풍선 타이핑 효과가 이를 얼마나 완화하는지, 타자 속도가 단어 길이에 비해 빠르게 느껴지는지 관찰
- **ASCII 전환 중 성능**: 4세션 누적 + 각 레이어에 opacity 2s 트랜지션 동시 작동. 모바일 저성능 기기에서 프레임 드롭 확인 필요
- **반전/공유 화면에 ASCII 배경이 남는 효과**: 의도된 연속성이지만, 반전 메시지("저는 사람이 아니에요")와 배경 ASCII가 시각적으로 경쟁할 수 있음. 필요하면 twist-screen 진입 시 컨테이너 opacity 감소 옵션 검토

## 세션 2 — 온보딩 호흡 보정 (프로필 로딩 / 인사는 문장 / 세션 경계는 시스템 알림)

### 맥락
첫 세션 변경 직후 플레이 감각 피드백 3가지:
1. 소개 입력 완료 → 규칙 화면이 "딸깍" 바로 뜸. 플레이어 입력이 어디로 갔는지 호흡이 없음
2. 파트너 첫 인사까지 단어 배열로 나오니, 게임 규칙이 적용되기도 전부터 이미 "단어만 쓰는 사이"처럼 읽힘 — 상대가 기계처럼 느껴짐
3. 세션 종료 시 "계속할까요?"가 파트너 말풍선으로 나와서, 상대가 게임 진행자를 겸하는 어색한 포지션이 됨

### 주요 변경 (`index.html`)

**1) 프로필 전송 로딩 화면 (`sending-screen`)**
- intro-screen 완료 → `playSendingProfile()` (1.6초 progress bar + 0.5초 여백) → rules-screen
- 매칭 화면과 동일한 바/커서 UI 재사용, 문구만 `"프로필을 보내는 중"`으로 차별화
- 실제 네트워크 호출은 없음 (의도적 가짜 딜레이). AI 정체 은폐 원칙에 맞춰 "서버·API" 같은 기술 용어는 문구에서 배제

**2) 첫 인사 문장 복원 (`addPartnerSentence`)**
- `greeting_words` → `greeting` 문자열로 롤백. `FALLBACK_GREETING_WORDS` → `FALLBACK_GREETING = '잠깐 이야기하다가 세계 만들어볼까요'`
- state: `firstGreetingWords` 제거, `firstGreeting` 문자열 복원
- 프롬프트에 "세계 만들기가 시작되면 단어로만 소통하지만, 지금은 시작 전"이라고 명시해 문장 인사만 나오도록 유도
- 렌더링: partner-screen은 원래부터 typewriter로 단순 문자열을 찍어주니 그대로. 세션 첫 진입의 채팅 첫 메시지는 `addPartnerSentence` (테두리 없음, 본문 폰트)로 바꿔, 이후 등장하는 단어 말풍선(`.chat-msg.partner` 기본: 테두리 + Mono)과 시각적으로 구분
- 단어 규칙 적용 시점: **세계 만들기가 시작된 순간부터**. 그 이전의 인사만 문장

**3) 세션 종료 경계를 시스템 알림으로 (`addSystemMsg`)**
- 파트너 말풍선 `addPartnerWords(['계속', '할까요'], ...)` 제거
- `.chat-msg.system` CSS 신규: 중앙 정렬, 흐린 색, Mono, letter-spacing 0.15em, 상하 얇은 선(`border-top`/`border-bottom`)
- 문구: `'계속할까요?'` (중간 세션) / `'다 만들었어요'` (마지막 세션)
- 채팅 흐름이 "파트너 대화 → 시스템 알림 → 버튼" 순으로 계층이 분리됨. 파트너는 더 이상 진행자 역할을 겸하지 않음

### 결정 사항
- **로딩 화면에 "프로필을 보내는 중" 문구**: "전송", "매칭 준비 중" 등 여러 후보 중 선택. "서버"나 "API"는 금지어이지만 "보내는 중"은 일반 구어라 AI 은폐 원칙과 충돌 없음. 매칭 단계 문구("파트너를 찾는 중")와 라임이 맞아 같은 톤
- **인사만 문장, 나머지 대화는 단어 유지**: 규칙을 본 직후에 규칙이 즉시 작동하기 시작하는 내러티브 — 플레이어 입장에서 "아, 세계 만들기가 진짜 시작됐구나"를 인지하는 트리거. 첫 인사가 단어였던 이전 버전은 규칙 학습 타이밍과 실제 적용 타이밍이 겹쳐 구분이 흐릿했음
- **`FALLBACK_GREETING` 폴백 문구**: 단어 2개짜리에서 문장 한 줄로 갈아치움. 실패 시에도 파트너가 "사람처럼" 말을 거는 톤 유지
- **"계속할까요?"는 물음표 유지, "다 만들었어요"는 마침표 없이 짧게**: 시스템 알림이지만 본문은 구어로 두어 차가움 완화. 이 공간이 여전히 두 사람 사이의 순간이라는 느낌
- **시스템 알림 스타일에 상하 경계선**: 단순 흐린 텍스트보다 "세션이 끊어졌다"는 시각적 구분이 강함. 대화 블록과 버튼 영역 사이의 경계로 기능
- **로딩 화면은 1.6초**: 너무 짧으면 무의미, 너무 길면 지루. 매칭 바(2.5초)보다 짧게 해 "프로필 검증 → 규칙 안내 → 매칭"의 호흡 계단이 점점 길어지도록 설계

### 미해결
- 첫 인사 문장이 규칙 화면의 "단어로만 소통" 규칙과 모순되어 보일 가능성. 프롬프트에 "지금은 시작 전"이라 명시했지만 모델이 혼동해 단어 배열로 회귀하면 파싱은 실패할 수 있음 → `FALLBACK_GREETING`로 자연 복구
- 세션 종료 시스템 알림의 한국어 letter-spacing이 실기기에서 너무 넓거나 좁아 보일 가능성. 실플레이 관찰 후 조정

## 세션 3 — 멀티 턴 대화 + AI 주도 세션 종료

### 맥락
이전 버전은 세션당 단어 1턴만 주고받으면 시스템 알림이 바로 "계속할까요?"로 넘어가 대화감이 전혀 없었다. 플레이어는 확정 의사도 낼 수 없고, 파트너도 세션을 "언제 끝낼지" 결정할 권한이 없어 두 사람이 말 맞추는 의식(ceremony)이 부재했다. 멀티 턴과 양측 확정 시그널을 도입해 "충분히 이야기했다"는 감각을 게임 메커니즘으로 옮김.

### 주요 변경 (`index.html`)

**1) state 확장 + 세션 진입 초기화**
- `state.turnCount` / `state.sessionKeywords` / `state.sessionChatHistory` 추가
- `enterSession(layerIndex, firstEntry)` 진입 시 세 필드 전부 리셋 + `hideConfirmBtn()`
- `new-world-btn` 리셋에도 동일하게 초기값

**2) "이걸로 할게요" 확정 버튼 (`.confirm-btn`)**
- `.input-area` 내부, send-btn 옆에 배치. 기본 `opacity: 0; pointer-events: none`, `.visible` 클래스로 0→1 페이드인(0.5s)
- `showConfirmBtn()` / `hideConfirmBtn()` 헬퍼
- 노출 조건: ① `session_signal === 'propose_end'` 이거나 ② `state.turnCount >= 2` (continue여도 플레이어가 끊을 권한). disagree 이후엔 다시 숨김

**3) Gemini 응답에 `session_signal` 도입**
- `callGemini(layer, words, onStall, confirm)` 시그니처에 `confirm` 추가. 프롬프트를 분기해 두 개의 스키마를 낸다.
  - 일반 턴(`confirm=false`): `{ partner_words, session_signal: 'continue'|'propose_end' }` — narrative/ascii_art/mood 없음, 토큰 절약
  - 확정 턴(`confirm=true`): `{ partner_words, session_signal: 'agree'|'disagree', narrative, ascii_art, mood }`
- 프롬프트에 판단 기준을 한국어로 삽입: propose_end는 30% 이내, disagree는 20% 이내 유도
- `sessionChatHistory`를 프롬프트에 풀어 넣어 모델이 "턴 수 + 구체화 정도"를 맥락으로 판단할 수 있게 함
- `tryJsonOnce` 검증 키는 두 분기 공통인 `['partner_words', 'session_signal']`만. narrative/mood/ascii_art 존재 여부는 클라이언트가 agree 시점에 다시 검증

**4) `submitWords` 멀티 턴화**
- `addMyWords` → `sessionKeywords.push(...)` → `sessionChatHistory.push({who:'me', ...})` → `turnCount++`
- confirm=false로 `callGemini` → `addPartnerWords`
- session_signal 분기:
  - `propose_end`: `addSystemMsg('이 정도면 됐을 것 같은데요')` + `showConfirmBtn()`
  - `continue` + turnCount ≥ 2: `showConfirmBtn()`만 (알림 없이 슬며시)
  - 1턴째 continue: 아무것도 안 함
- 자동 `showSessionEndChoice()` 호출 제거 → 세션은 agree에서만 닫힘

**5) `handleConfirm` 신설 (확정 클릭 처리)**
- 대표 키워드: `sessionKeywords`를 중복 제거 후 앞 3개 (없으면 방어적 early return)
- `addMyWords(rep)`로 대표 단어 말풍선 + `sessionChatHistory`에 `confirm: true`로 마킹
- confirm=true로 `callGemini`
- `signal === 'agree' && narrative && mood && ascii_art` → `applyMood` + `addAsciiLayer` + `state.sessions.push` + `saveWorld` + `renderWorld` + 1초 대기 → `showSessionEndChoice`
- 그 외(disagree / agree인데 필드 누락) → 입력 재활성화, 대화 재개. 확정 버튼은 숨긴 상태 유지 (다음 턴부터 자연스럽게 다시 등장)

**6) `showSessionEndChoice` 간결화**
- 내부 `addSystemMsg('계속할까요?' / '다 만들었어요')` 제거
- agree로 받은 파트너 단어 자체가 이미 세션 종료 맥락을 전달하므로 시스템 알림은 중복

**7) `showPartnerStallRetry` 분리**
- 기존 submitWords 안에 인라인돼 있던 파트너 실패 UI를 별도 함수로 뽑음. submitWords/handleConfirm 모두에서 재사용 가능하게

### 결정 사항

- **narrative/mood/ascii_art는 agree 시점에만 생성**: 매 턴 받으면 토큰 낭비 + 세계가 매 턴마다 덮어쓰기돼서 "레이어 1개 = 세계 1층" 개념이 무너짐. 확정 시점이 곧 "세계에 각인되는 순간"이라는 의식적 구분을 만듦
- **일반 턴 스키마를 별도로 만들어 프롬프트 길이 감축**: flash-lite에 매 턴 30×15 ASCII 명세를 주면 instruction following이 흔들림. 필요할 때만 full schema를 요구
- **`session_signal` 판단을 전적으로 모델에 위임**: 클라이언트가 "turnCount가 4 이상이면 무조건 propose_end" 같은 하드코딩을 하지 않음. 모델이 문맥을 보고 결정해야 파트너가 '사람 같은' 타이밍 감각을 가진다
- **propose_end 30% / disagree 20% 상한을 프롬프트에 명시**: 상한 없으면 lite가 매 턴 propose_end로 플레이어를 몰아가는 경향이 있었을 것. 명시적 비율 제약이 분포 조절에 효과적
- **2턴 이상 continue에서도 버튼 노출**: 파트너가 끝낼 생각이 없을 때 플레이어가 주도할 권한. AI가 끝없이 "계속"하는 상황을 플레이어가 끊을 수 있어야 함
- **disagree 시 버튼 자동 재노출 안 함**: "파트너가 더 하자고 함" → 대화가 한 턴 더 쌓여야 자연스럽게 다시 확정 가능. 즉시 재노출하면 플레이어가 같은 버튼을 연타해 AI의 거절을 무시하는 느낌이 됨
- **확정 시 대표 키워드는 중복 제거 후 앞 3개**: 세션 초반에 던진 단어가 보통 주제어. 마지막 3개나 빈도순도 후보였지만 "세션의 출발점"을 각인하는 의미에서 앞 3개가 맞음. 사용자가 세션 중반에 방향을 꺾어도 최종 말풍선은 원래 주제를 보존
- **agree 필드 누락 시 disagree로 격하**: lite가 session_signal=agree로 답했지만 narrative를 빈 문자열로 비우는 케이스가 예상됨. 세션을 잘못 닫는 것보다 대화를 한 턴 더 여는 쪽이 복구 비용이 낮음
- **state.sessions 저장 시 `allKeywords`/`chat` 함께 기록**: 대표 3개 키워드만 저장하면 세션에서 실제로 오간 대화가 사라져 공유 카드나 추후 기능에서 활용 불가. localStorage 비용은 미미

### 미해결

- **1턴에서 파트너가 propose_end 선언하는 경우 컨트롤**: 프롬프트 상 금지했지만 lite가 넘길 수 있음. 클라이언트에서 `turnCount >= 2`일 때만 propose_end 수락으로 하드 가드할지 검토
- **2턴 만에 agree가 너무 쉽게 나올 위험**: 프롬프트에 disagree 20% 상한을 줘서 대부분 agree에 쏠릴 수 있음. 실측 후 프롬프트에 "2턴만 하고 확정 요청이면 disagree 비율을 올리라" 같은 보정 필요할 수 있음
- **disagree 후 다음 submitWords의 버튼 노출 시점**: 현재는 다음 턴이 끝난 직후 등장. 플레이어 입장에선 "한 번 거절당하면 두 턴 더 해야 확정 가능"한 느낌. 의도 맞지만 과할 수 있어 관찰 필요
- **`addSystemMsg('이 정도면 됐을 것 같은데요')` 문구**: 시스템 알림이라기엔 1인칭 구어라 역할 경계가 애매. 실플레이에서 어색하면 `"파트너가 마무리를 제안했어요"` 같은 3인칭 명사형으로 재튜닝

## 세션 4 — ASCII 대신 Canvas 배경, 인사·확정 단어 하드코딩, 퀘스트 힌트

### 맥락
ASCII 아트를 Gemini가 매 세션 30×15로 생성하게 했더니 프롬프트가 비대해져 lite 모델이 instruction을 놓쳤고, 무엇보다 429 quota 폭발 원인. 동시에 매 세션 greeting/partner_words/narrative/mood/ascii_art 다섯 개 필드를 모두 요청하는 구조가 과했다. "세계에 각인되는 것"만 API로 받고 나머지는 클라이언트에서 처리하는 방향으로 정리.

### 주요 변경 (`index.html`)

**1) API 호출 축소**
- `buildGreetingPrompt` / `fetchGreeting` / `FALLBACK_GREETING` 전부 삭제. `PARTNER_GREETINGS` 6개 후보 중 랜덤 선택으로 교체. `introducePartner`는 600~1400ms `sleep`만 두고 바로 선택
- `buildPrompt`에서 `ascii_art` 필드·`priorAscii` 블록·ascii 작성 규칙 모두 제거
- 확정 턴(`confirm=true`) 프롬프트에서는 `partner_words`도 요청하지 않음. `session_signal`과 agree 시 `narrative`·`mood`만 받음
- `callGemini`의 `required` 키가 confirm에 따라 분기: 일반 `['partner_words','session_signal']`, 확정 `['session_signal']`
- 결과: 한 세션당 Gemini 호출 수는 그대로지만 토큰은 약 40~50% 감소, 파싱 실패 위험도 함께 감소

**2) Canvas 기반 mood 배경 엔진 (Obra Dinn 느낌)**
- `.ascii-layer`(div + 문자열) → `.mood-layer`(canvas) 교체. 누적·opacity 로직은 그대로 (최신 1.0 / 0.35 / 0.2 / 0.1)
- `drawMoodBackground(mood)`:
  - `hexToRgb`/`desaturate(factor=0.55)`로 primary_color를 중간회색 쪽으로 절반 블렌드 → 1비트 톤 느낌
  - `pickPattern(mood.keywords)`로 6개 패턴 중 선택: water(수평 물결, 하단 농도↑) / forest(랜덤 수직선, 상단 밀도↑) / city(격자 + 랜덤 블록) / fog(반투명 radial gradient 다수) / fire(하단 중심 방사형) / dots(랜덤 점)
  - 각 패턴은 단색 + `globalAlpha` 변동만으로 명암을 만들어 흑백 판화 인상
  - CSS `filter: contrast(120%)` + `transition: opacity 2s ease`로 페이드인 + 고대비
- `PATTERN_KEYWORDS` 테이블은 부분 문자열 매칭(`kw.includes(m) || m.includes(kw)`)이라 "푸른 바다", "안개 낀 길" 같은 복합어도 수용

**3) 확정 흐름 하드코딩**
- 플레이어 확정 말풍선 = `CONFIRM_MY_WORDS = ['여기까지','다음']` 고정
- agree 시 파트너 단어 = `CONFIRM_PARTNER_AGREE` 4개 풀 랜덤
- disagree 시 파트너 단어 = `CONFIRM_PARTNER_DISAGREE` 3개 풀 랜덤
- Gemini가 반환하는 `partner_words`는 확정 턴에서는 무시 (애초에 요청 안 함)
- `session_signal`만 모델 판단에 맡기고, 그에 따른 문구는 클라이언트 결정 — 파트너의 "톤"은 하드코딩으로 고정, "결정권"은 모델에 위임

**4) 퀘스트 인디케이터 (`#quest-hint`)**
- 입력창 바로 위, JetBrains Mono 0.82rem, dim 색
- 3단계 텍스트: `early(layer)` = "${layer}를 단어로 묘사해보세요 (1-5개)" / `ready(layer)` = "계속 단어를 나눠도 되고, 마음에 들면 확정할 수 있어요" / `propose()` = "파트너가 확정을 제안했어요"
- 호출 타이밍: enterSession=early / 턴 완료 후 turnCount≥2=ready / propose_end 응답=propose / disagree 후=ready / showSessionEndChoice 진입 시 숨김

**5) 시스템 알림 완전 제거**
- `addSystemMsg`와 `.chat-msg.system` CSS 삭제. 이전엔 propose_end/세션 종료에서 썼는데, 퀘스트 힌트가 그 역할을 더 깔끔하게 전담
- showSessionEndChoice에는 이미 알림이 제거돼 있었음. 남은 한 곳(submitWords의 propose_end) 도 퀘스트 힌트로 대체

### 결정 사항

- **greeting을 완전히 하드코딩으로 전환**: 6개 후보면 충분히 다양하게 보이고, 세션 시작에서 API 실패로 멈추는 리스크가 0이 됨. 개성은 `PERSONAS` tone이 세션 중에 드러나므로 첫 인사에서 고유성을 살릴 필요 없음
- **`ascii_art`를 빼면서 배경 자체도 더 좋아짐**: LLM이 텍스트로 그린 30×15는 모델 품질 편차가 커 "느낌"이 불안정했음. Canvas 패턴은 결정론적이고 mood 키워드만 맞추면 일관된 분위기 생성. 렌더링도 더 부드러움
- **Obra Dinn 톤(`desaturate 0.55 + contrast 120%`)**: 원색 hex 그대로 쓰면 UI가 만화처럼 보임. 중간회색에 절반만 섞어 채도를 낮추고 CSS contrast로 명암만 과장 → "인쇄된 듯한" 질감
- **확정 말풍선의 하드코딩은 "대표 키워드 반복" 문제 해결**: 이전엔 플레이어가 세션 중 던진 단어 3개를 다시 찍었는데, 직전 말풍선과 시각적으로 중복되고 확정의 '의식적 의미'가 약했음. "여기까지 / 다음"은 단순한 닫는 제스처라서 맥락 전환이 더 또렷
- **파트너 확정 단어도 하드코딩**: Gemini가 agree/disagree 각각에 적절한 단어를 매번 잘 뽑는 게 어렵고 실패 시 어색함이 컸음. 풀에서 고르면 일관된 '리액션' 톤을 보장하면서도 4~3개 풀이라 반복감은 적음
- **`session_signal`은 여전히 모델에 위임**: 하드코딩으로 다 밀어버리지 않은 이유. "언제 끝낼지"는 대화 내용·턴 수 맥락을 LLM이 판단해야 파트너가 '사람 같은 판단력'을 가진 것처럼 보임. 단어만 포장을 바꾼 셈
- **퀘스트 힌트가 시스템 알림을 대체**: 알림은 채팅 흐름을 끊는 요소였고, 힌트는 입력창 위에 상주해서 플레이어가 "지금 뭘 할 수 있는지" 항상 볼 수 있음. 인지 부담 감소
- **힌트 문구에 느낌표·마침표 없음**: 대화 UI의 느슨한 안내라 문장 부호를 최소화

### 미해결
- Canvas 패턴이 6개뿐이라 mood.keywords가 전부 매칭 안 되면 `dots`로 귀결. 실플레이에서 "점만 찍힌 배경"이 자주 나오면 기본 패턴을 좀 더 분위기 있게 바꾸거나 패턴 카테고리 추가 필요
- `desaturate 0.55`가 primary_color에 따라 너무 회색으로 수렴할 수 있음. 파랑계열은 괜찮지만 노랑·주황은 밋밋해질 위험
- 확정 파트너 풀이 적어서 두 세션 이상 하면 같은 조합이 반복될 가능성. 풀 확장 or 중복 방지 로직 고려
- 퀘스트 힌트 `ready`/`propose` 전환이 순식간에 일어나면 플레이어가 변경을 못 볼 수 있음. 글자 교체 시 0.3s fade 정도 넣을지 관찰 후 결정
